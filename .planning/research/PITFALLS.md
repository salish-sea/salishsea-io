# Pitfalls Research

**Domain:** Producing a Darwin Core Archive (DwC-A) from a multi-source whale-observation database (download-only, not GBIF-registered)
**Researched:** 2026-06-09
**Confidence:** HIGH for DwC-A structural rules and GBIF validator flags (authoritative GBIF/TDWG docs); MEDIUM for operational nightly-file and rights-aggregation strategies (verified against GBIF guidance + general practice); HIGH for data-model gaps (read directly from this project's schema).

---

## How to read this file

Three distinct severities are kept separate on purpose, because the downstream roadmapper needs to triage them differently:

- **BLOCKER** — the archive will not load / fails structural validation. Must be fixed or the deliverable does not exist. (DwC-A structural rules.)
- **QUALITY** — the archive loads but consumers (GBIF/OBIS validators, researchers) flag the records as low-quality or misinterpret them. Matters because the audience *is* researchers; degrades the value of the deliverable.
- **RIGHTS** — legal/ethical correctness of redistributing third-party and contributor data. Independent of whether the file validates. Getting this wrong is the most expensive to recover from (reputational, and may require re-issuing the archive).
- **GAP** — a data-model deficiency that must become an explicit, surfaced finding/requirement, NOT silently fudged. The milestone goal literally says "identify actual gaps in data and datatype." These are called out as their own category at the end.

Grounding note: pitfalls reference this project's real `public.occurrences` view (see `supabase/migrations/20260204175500_admins.sql`), where Maplify uses `created_at AT TIME ZONE 'GMT'` as `observed_at`, `accuracy` is `null` for Maplify, native `public.observations` has **no record-level license column**, and photo licenses live per-photo on `sighting_photos.license_code` / `inaturalist.observation_photos.license`.

## Critical Pitfalls

### Pitfall 1: meta.xml field-index ↔ column misalignment (BLOCKER)

**What goes wrong:**
`meta.xml` maps each data column to a Darwin Core term by **0-based positional index**, not by header name. If the index attributes drift out of sync with the actual column order in the TSV (someone reorders a `SELECT`, adds a column, or hand-edits `meta.xml`), every record loads with values shifted into the wrong terms — `eventDate` lands in `decimalLatitude`, etc. The archive often still "validates structurally" because indices are in range; the corruption is silent.

**Why it happens:**
The TSV is generated by SQL and `meta.xml` is authored separately. The two are coupled by ordinal position with no compile-time link. The `<id index="0">` / `<coreid index="0">` and each `<field index="N" term="...">` must exactly match the emitted column order.

**How to avoid:**
- Generate `meta.xml` and the TSV **from a single ordered field list** (one source-of-truth array of `{index, dwcTerm, sqlExpression}`), so column order and `meta.xml` indices cannot diverge. Never maintain them independently.
- Set `ignoreHeaderLines="1"` and still write a header row, then add a test that asserts the header names equal the `term` order in `meta.xml`.
- Round-trip test: parse the produced archive back (e.g. with a DwC-A reader or python-dwca-reader) and assert a known record's `occurrenceID` maps to the expected `decimalLatitude` value.

**Warning signs:**
Coordinates that parse as dates, scientificName containing numbers, validator reporting impossibly many `COORDINATE_INVALID` at once.

**Phase to address:** Archive-generation phase (the phase that emits meta.xml + core/extension files).

### Pitfall 2: Core↔extension coreId join breakage for multimedia (BLOCKER / silent data loss)

**What goes wrong:**
Photos are a separate Multimedia extension file linked to the Occurrence core by `coreId == occurrenceID`. If the extension's `coreid` value doesn't byte-for-byte match a core `id` (whitespace, differing prefix like `maplify:` vs bare id, different escaping), the photos silently attach to nothing — they neither error nor appear. With this project's IDs being prefixed strings (`maplify:244458`, `inaturalist:...`, native UUIDs), prefix consistency across files is the failure surface.

**Why it happens:**
The core and extension files are produced by separate queries; the join key is reconstructed in each and can drift. Multimedia rows for excluded sources (iNat/Happywhale) or for occurrences filtered out of the core leave orphan coreIds.

**How to avoid:**
- Derive `occurrenceID` once, in one expression, and reuse it verbatim for both core `id` and extension `coreid`.
- After generation, assert every extension `coreid` exists in the set of core `id`s (anti-join must be empty), and that no core row is dropped because its photos query failed.
- Apply the **same source-exclusion filter** (exclude iNaturalist + Happywhale) to the multimedia query as to the core query.

**Warning signs:**
Photo count in extension ≠ expected; validator "extension records referencing non-existent core records"; downloads where occurrences that should have images show none.

**Phase to address:** Archive-generation phase.

### Pitfall 3: occurrenceID non-uniqueness or instability (BLOCKER + long-term trap)

**What goes wrong:**
DwC requires `occurrenceID` (the core `id`) to be **unique within the archive** and, for any future GBIF registration, **stable across re-publications**. Two failure modes: (a) duplicate IDs in one nightly file → archive rejected; (b) IDs that change night-to-night (e.g. derived from a row number or regenerated surrogate) → if ever registered, GBIF treats every record as new each night, destroying record history and inflating counts.

**Why it happens:**
The unified `occurrences` view already namespaces IDs (`'maplify:' || s.id`, `'inaturalist:' || id`, UUIDs for native). Risk arises if native observation IDs or a synthetic export ID are introduced that aren't globally unique across the UNION, or if dedup across overlapping sources isn't considered.

**How to avoid:**
- Keep the existing source-prefixed scheme as the `occurrenceID`; it is already namespaced and stable. Document it as the stable identifier contract.
- Add a uniqueness assertion over the emitted core `id` column (GROUP BY ... HAVING count > 1 must be empty) as a generation gate.
- Even though registration is deferred, treat IDs as a public contract now (the milestone says "kept reachable by design"). Do not derive IDs from row position, sequence, or generation timestamp.

**Warning signs:**
Generation-time uniqueness check fails; same sighting appearing under two IDs.

**Phase to address:** Data-model alignment phase (decide the occurrenceID contract) + generation phase (enforce it).

### Pitfall 4: CSV/TSV quoting, escaping, embedded newlines, and BOM (BLOCKER)

**What goes wrong:**
Freeform `body` text (and `attribution`) contains commas, double-quotes, `<br>` HTML, emoji (♂ ♀ 📍 from the Happywhale branch — though excluded here, native/Maplify bodies still carry arbitrary text), and literal newlines. If the export uses naive string concatenation, an embedded delimiter or newline splits one logical record into several physical rows → field-index shift for every subsequent column, cascading corruption. A UTF-8 BOM prepended to the first file makes the first column header (`id`) unrecognized by strict parsers.

**Why it happens:**
Developers hand-roll TSV writing and forget that the field/line terminators declared in `meta.xml` (`fieldsTerminatedBy`, `linesTerminatedBy`, `fieldsEnclosedBy`) impose escaping obligations: any field containing the delimiter, the enclosure char, or a line terminator MUST be enclosed, and enclosure chars inside must be doubled. TSV is *not* safe just because tabs are rare — newlines and quotes still occur.

**How to avoid:**
- Use a real CSV/TSV serializer (or PostgreSQL `COPY ... WITH (FORMAT csv, ...)`) that honors `fieldsEnclosedBy='"'` and doubles embedded quotes — do **not** concatenate strings.
- Declare the exact terminators/enclosure in `meta.xml` and make the serializer config match them literally.
- Strip or normalize: convert `<br>` and HTML in `body` to plain text/newlines *before* serialization, normalize CRLF→LF (or pick one and declare it in `linesTerminatedBy`), and write UTF-8 **without BOM**.
- Test with a record that deliberately contains a comma, a `"`, a newline, and a multibyte character; round-trip and assert field integrity.

**Warning signs:**
Record count in parsed archive ≠ row count in query; a stray `"` mid-file; mojibake in viewers; first header reads `<U+FEFF>id`.

**Phase to address:** Archive-generation phase (and a body-text-normalization sub-task, since HTML stripping touches the partner-org linkifier from v1.1).

### Pitfall 5: eventDate fabricated from Maplify report time; precision overstated (QUALITY + GAP)

**What goes wrong:**
Native observations have a true sighting time (`observed_at timestamptz`). Maplify records only have `created_at` (a **report/submission** time), which the view already coerces via `created_at AT TIME ZONE 'GMT'` into `observed_at`. Emitting that as a full ISO-8601 datetime `eventDate` asserts a precision the data does not have — it claims the whale was seen at the moment the report was filed. Researchers downstream will treat it as observation time. GBIF won't *reject* it, but it's a correctness lie.

**Why it happens:**
The unified view papers over the source difference so the map "just works." Export inherits the fudge. ISO-8601 also *permits* reduced precision (`2026-06-09` or `2026-06`), but naive formatting always emits full datetime.

**How to avoid:**
- **Surface this as an explicit GAP** (see GAP-A), don't silently emit report-time as sighting-time.
- Where source = Maplify and only report time exists, consider emitting `eventDate` at **date precision** (`YYYY-MM-DD`) rather than a false second-level timestamp, and/or record the report time in `eventRemarks`/`dataGeneralizations` and use `eventTime` only when genuinely known.
- Format eventDate as ISO-8601 with no leading-zero padding issues and ensure year/month/day components (if also emitted) agree with the string to avoid `RECORDED_DATE_MISMATCH`.
- Convert all timestamps to a single explicit timezone offset (UTC `Z`) deterministically; don't rely on server-local TZ.

**Warning signs:**
GBIF `RECORDED_DATE_MISMATCH` or `RECORDED_DATE_INVALID`; a researcher noticing every Maplify sighting is at a suspiciously round/clustered time.

**Phase to address:** Data-model alignment phase (decide per-source temporal precision policy) → generation phase (implement). Must be a documented finding.

### Pitfall 6: coordinateUncertaintyInMeters = 0 (or defaulting unknown to 0) (QUALITY/BLOCKER-for-GBIF + GAP)

**What goes wrong:**
GBIF flags `COORDINATE_UNCERTAINTY_METERS_INVALID` when the value is not a plausible number, and **0 is explicitly invalid** (valid range is >0 and <5,000,000 m). Maplify accuracy is `null` in the view (`null AS accuracy`). The tempting fix — default unknown accuracy to `0` — is precisely the invalid value, and worse, semantically claims *perfect* precision for the least-precise records.

**Why it happens:**
A non-null numeric field feels "cleaner" than null; 0 looks like a harmless default. It is the single most common coordinate-uncertainty error.

**How to avoid:**
- **Leave `coordinateUncertaintyInMeters` empty (omit) when unknown** — never 0. An absent value is valid; 0 is not.
- For sources with known accuracy (native may capture EXIF/GPS; iNat has `public_positional_accuracy`), map the real meters value.
- Surface "Maplify has no positional accuracy" as a GAP (see GAP-B) rather than inventing a number. If a conservative coarse estimate is desired, choose a defensible positive value and document it in `dataGeneralizations`/`georeferenceRemarks` — but prefer omission.

**Warning signs:**
Validator `COORDINATE_UNCERTAINTY_METERS_INVALID`; any literal `0` in the column.

**Phase to address:** Data-model alignment phase (policy: omit-when-unknown) → generation phase.

### Pitfall 7: Missing geodeticDatum, and decimalLatitude/Longitude rounding/sign errors (QUALITY)

**What goes wrong:**
Three sub-traps:
1. **No `geodeticDatum`** → GBIF assumes WGS84 (`GEODETIC_DATUM_ASSUMED_WGS84`, a warning). Data *is* WGS84 (`geography(Point,4326)`), so this is easy to state explicitly and silence the flag.
2. **Sign/axis swap** — extracting lon/lat from PostGIS: `ST_X` is longitude, `ST_Y` is latitude. The view's `lon_lat` row is `(ST_X, ST_Y)`. Mapping `decimalLatitude`←X or `decimalLongitude`←Y silently puts every Salish Sea whale in the wrong hemisphere/ocean.
3. **Over/under-rounding** — emitting too few decimals collapses distinct points; GBIF rounds to 6 decimals (`COORDINATE_ROUNDED`) and excessive precision is meaningless. ~6 decimals (~0.1 m) is the sweet spot.

**How to avoid:**
- Emit a constant `geodeticDatum = "EPSG:4326"` (or `"WGS84"`) for all records.
- Unit-test the lat/lon mapping with a known coordinate (e.g. a point near 48.5°N, -123°W) asserting latitude is positive ~48 and longitude is negative ~-123.
- Format coordinates with ~6 decimal places, preserving sign (Salish Sea longitudes are negative).

**Warning signs:**
Points plotting on land, in the southern hemisphere, or off the coast of Africa (0,0); `COORDINATE_OUT_OF_RANGE`.

**Phase to address:** Generation phase, with a coordinate-mapping unit test (the codebase already has fragile-projection concerns per CONCERNS.md — reuse that scrutiny here).

### Pitfall 8: taxonRank vs scientificName mismatch; genus/family-only IDs (QUALITY + model alignment)

**What goes wrong:**
Some occurrences are identified only to genus or family (the `taxa` hierarchy includes ranks from species up to kingdom). If `scientificName` is emitted but `taxonRank` is wrong or absent, or if a genus name is emitted with `taxonRank=species`, GBIF flags `TAXON_MATCH_HIGHERRANK` or mismatches the backbone. Also, emitting a binomial when only genus is known overstates the identification.

**Why it happens:**
The view emits `scientific_name` and `vernacular_name` but the export must also walk the `inaturalist.taxa` parent chain to populate `taxonRank`, `kingdom`, `phylum`, `class`, `order`, `family`, `genus`. Forgetting `taxonRank`, or assuming everything is a species, breaks higher-taxon records.

**How to avoid:**
- Emit `taxonRank` from `inaturalist.taxa.rank` for every record (it exists — it's the enum in the schema).
- Walk the parent hierarchy to fill the Linnaean classification fields (this is an explicit milestone target — "walk the `taxa` hierarchy to kingdom→genus").
- Always emit `kingdom` (Animalia) — it's cheap and improves backbone matching.
- For genus/family-only records, set `scientificName` to the actual rank's name and `taxonRank` accordingly; do not synthesize a species name.

**Warning signs:**
`TAXON_MATCH_HIGHERRANK`, `TAXON_MATCH_NONE`, records where a family name carries species rank.

**Phase to address:** Data-model / taxonomy-alignment phase (the hierarchy walk) → generation phase.

### Pitfall 9: occurrenceStatus and individualCount conflicts (QUALITY)

**What goes wrong:**
`occurrenceStatus` should be `present` for these sightings. Counts can be **null** (Maplify uses a range; native `count` may be null) and the view already nulls counts outside 1–1000. If `individualCount=0` is ever emitted while status is `present`, GBIF flags `INDIVIDUAL_COUNT_CONFLICTS_WITH_OCCURRENCE_STATUS`. Conversely, omitting `occurrenceStatus` entirely makes GBIF infer it from count (`OCCURRENCE_STATUS_INFERRED_FROM_INDIVIDUAL_COUNT`).

**How to avoid:**
- Hard-set `occurrenceStatus = "present"` for all exported records (these are all positive sightings; there are no absence records).
- Emit `individualCount` only when a real positive count exists; **omit (leave blank) when null** — never 0.
- Never emit `individualCount=0`.

**Warning signs:**
`INDIVIDUAL_COUNT_CONFLICTS_WITH_OCCURRENCE_STATUS`, `INDIVIDUAL_COUNT_INVALID`.

**Phase to address:** Generation phase.

### Pitfall 10: basisOfRecord guessed or omitted (QUALITY + GAP)

**What goes wrong:**
`basisOfRecord` is required and controlled-vocabulary. Omitting it or inventing a value triggers `BASIS_OF_RECORD_INVALID` and the record may be dropped. For community whale sightings the correct value is `HumanObservation`. The trap is defaulting to something like `Occurrence` or `MachineObservation`, or leaving it blank.

**How to avoid:**
- Set `basisOfRecord = "HumanObservation"` for native + Maplify/Whale Alert records (all are human-reported sightings).
- If any source were sensor/acoustic-derived it would be `MachineObservation` — confirm none of the in-scope sources are, and document the decision.

**Warning signs:** `BASIS_OF_RECORD_INVALID`; records silently absent from a GBIF preview.

**Phase to address:** Data-model alignment phase (assign per-source basisOfRecord; document) → generation phase.

### Pitfall 11: Redistributing third-party (Maplify/Whale Alert) data without rights clarity (RIGHTS — highest recovery cost)

**What goes wrong:**
Maplify/Whale Alert records (and their nested "[Orca Network]", "Cascadia Trusted Observer", "Point Blue" provenance visible in `occurrence-bodies.tsv`) are third-party data being baked into a redistributable, downloadable archive under SalishSea.io's banner. If the source's terms forbid redistribution, or require attribution that the archive drops, the project is republishing others' data without a license to do so. GBIF requires a dataset-level CC license (CC0 / CC-BY / CC-BY-NC); choosing one implicitly asserts you have the right to license the contents — which you may not for aggregated third-party records.

**Why it happens:**
The unified view makes Maplify records look native. The `attribution` string (`usernm || ' on ' || source`) is the only provenance carried, and freeform "[Orca Network] ... (Joe Dreimeller)" attribution is buried in `body`, not in a structured rights field.

**How to avoid:**
- **Before publishing, confirm redistribution rights** for Maplify/Whale Alert data. This is a requirements/legal question, not a coding one — surface it as a blocking decision (GAP-D).
- Populate structured DwC rights terms per record: `rightsHolder`, `dwc:datasetName` (e.g. "Maplify / Whale Alert"), `institutionCode`, and `bibliographicCitation`, so the original source is credited even within the aggregated file.
- Choose the **most restrictive** applicable dataset license (CC-BY-NC if any source requires non-commercial; otherwise CC-BY). Per GBIF guidance, a restrictive source license effectively constrains the whole aggregate.
- Consider scoping the first archive to **native observations only** if third-party rights are unresolved, with Maplify added once cleared — keeps the deliverable unblocked.

**Warning signs:**
No documented permission from Maplify/Whale Alert; attribution present only as freeform text; a source's ToS prohibiting bulk redistribution.

**Phase to address:** Requirements/rights phase — **must precede** generation. This is a gate, not a polish item.

### Pitfall 12: Missing native-observation record-level license (RIGHTS + GAP)

**What goes wrong:**
Native `public.observations` has **no license column** (confirmed: only `sighting_photos.license_code` exists per-photo; the occurrence record itself has no license field, and the `occurrences` view emits no record-level license). Contributors uploaded sightings without ever agreeing to a redistribution license. Publishing their observations and freeform body text under a CC license retroactively imposes terms they never accepted. Photos have per-photo licenses (cc0…cc-by-nc-nd or *none*) — photos with **no license must be excluded** from the multimedia extension (the iNat branch already filters `license IS NOT NULL`; native/Maplify photo licensing needs the same discipline).

**Why it happens:**
The schema predates any publishing intent. License was modeled at the photo level (for display attribution) but never at the observation/record level. Easy to assume "our data, our license."

**How to avoid:**
- **Surface the missing record-level license as a primary GAP** (GAP-C). Decide: add a `license` column to `public.observations`, set a default (e.g. CC-BY with contributor attribution) going forward, and decide how to treat historical records (grandfather under a stated default, or exclude until re-consented).
- Update the submission flow / ToS so new contributors agree to the chosen license at submission time.
- For multimedia: emit only photos that carry a permissive, explicit license; **omit (do not guess) license-less photos** to avoid asserting rights you don't have.
- Carry attribution into `recordedBy` / `rightsHolder` from the `attribution` field.

**Warning signs:**
Export code that hard-codes a single license for all native records without a column backing it; license-less photos appearing in the multimedia file.

**Phase to address:** Data-model alignment phase (add license column + policy) + a contributor-consent decision in the requirements phase.

### Pitfall 13: Unvalidated whale identifiers leaking as authoritative occurrenceID/organismID (QUALITY + GAP)

**What goes wrong:**
Whale identifiers like `T065S` are **regex-extracted from freeform body text** (`extract_identifiers`), never validated against a catalog. If these are emitted as `organismID` or `associatedOccurrences` they assert verified individual identity that doesn't exist — and the milestone explicitly **defers** organismID linkage. Emitting them anyway would mislead researchers.

**How to avoid:**
- Do **not** emit regex-extracted identifiers as `organismID` this milestone (consistent with the deferral decision in PROJECT.md).
- If surfaced at all, place them in `dynamicProperties`/`occurrenceRemarks` clearly labeled as unverified text extraction, not in identity terms.

**Warning signs:** `organismID` populated from `identifiers`; researchers treating `T065S` as a confirmed match.

**Phase to address:** Generation phase (explicit exclusion) — already aligned with milestone scope.

### Pitfall 14: Nightly file written non-atomically; CloudFront serves stale/half-written zip (OPERATIONAL/BLOCKER-for-users)

**What goes wrong:**
A nightly regenerated public `.zip` has classic atomic-publish hazards:
- **Torn read:** a client downloads while the file is mid-upload → corrupt/truncated zip.
- **Stale cache:** CloudFront caches the old (or, worse, a half-written) object; users get yesterday's archive or a broken one long after regeneration.
- **Empty-result wipe:** a query glitch produces an empty/near-empty archive that overwrites a good one — silent data loss for downloaders.
- **Timezone ambiguity:** "nightly" in which timezone? An export keyed to server-local midnight vs Pacific vs UTC changes which day's records are "complete."

**Why it happens:**
S3+CloudFront is eventually consistent at the CDN edge; uploads aren't atomic from the client's perspective; no guard distinguishes "legitimately empty" from "broken."

**How to avoid:**
- **Write-then-swap:** upload to a temp key, verify (size, record count, zip integrity, validates), then atomically update a stable URL — either copy to the canonical key only after verification, or write versioned keys (`salishsea-dwca-2026-06-09.zip`) plus a stable `latest` pointer updated last.
- **Cache strategy:** set explicit `Cache-Control` and **invalidate CloudFront** for the canonical path after each successful publish (or use a content-hashed/dated filename so the URL changes and caching is never stale). Decide TTL deliberately (a daily file can have a short max-age).
- **Empty-result guard:** refuse to publish if record count is 0 or drops by more than a sane threshold vs the previous archive; alert instead of overwrite.
- **Define "nightly" explicitly** (recommend UTC or a fixed Pacific cutoff documented in the EML/README), and make the run idempotent.
- **Integrity:** publish a checksum (sha256) and/or size alongside, and ensure the zip is finalized before the pointer flips.

**Warning signs:**
Users report corrupt downloads; archive date in EML doesn't match content; an empty archive served; CloudFront serving a prior day after regeneration.

**Phase to address:** Hosting/operational phase (the phase that schedules + publishes the nightly artifact).

### Pitfall 15: Character encoding from freeform body text breaking the archive (OPERATIONAL/QUALITY)

**What goes wrong:**
`body`/`attribution`/`comments` contain HTML (`<br>`), emoji, accented names, and arbitrary Unicode. If any stage (Postgres client encoding, the serializer, the zip tooling, the EML) isn't strictly UTF-8 end-to-end, characters become mojibake or the file declares an encoding that doesn't match its bytes — breaking strict DwC-A readers and corrupting attribution.

**How to avoid:**
- Enforce UTF-8 at every stage: DB connection `client_encoding=UTF8`, serializer output UTF-8, `meta.xml encoding="UTF-8"`, no BOM.
- Strip/convert HTML in `body` to plain text before export (reuse/adapt the v1.1 partner-org link handling, but output text not HTML).
- Round-trip an accented name and an emoji through the full pipeline in a test.

**Warning signs:** Mojibake in `occurrenceRemarks`; reader errors on multibyte boundaries.

**Phase to address:** Generation phase (shared with Pitfall 4).

### Pitfall 16: Archive grows unbounded; nightly regeneration cost scales with full history (OPERATIONAL/PERF)

**What goes wrong:**
Each nightly run regenerates the **entire** archive from all history. As occurrences accumulate (years of sightings), full regeneration time, memory, and zip size grow. Combined with the client-side/perf concerns already noted in CONCERNS.md (no pagination, large-day loads), a naive `SELECT *` export will eventually time out the cron job or blow memory.

**How to avoid:**
- Stream the export (Postgres `COPY` to a stream → gzip/zip) rather than materializing all rows in memory.
- Right-size the runner (Supabase Edge Function vs a dedicated job) for full-history regeneration; benchmark against projected N-year row counts.
- Full regeneration nightly is acceptable at current scale; flag a future threshold where incremental/partitioned export is needed (document it, don't pre-build it).

**Warning signs:** Cron run time creeping up; memory pressure; archive size jumps.

**Phase to address:** Hosting/operational phase; note scale threshold as a documented future concern.

## Data-Model GAPS to surface as explicit findings (not fudge)

The milestone goal: *"Audit and document data/datatype gaps against DwC requirements; fill them where feasible."* These must appear as findings/requirements, **not** be silently defaulted.

| ID | Gap | The tempting fudge to AVOID | Correct handling |
|----|-----|------------------------------|------------------|
| GAP-A | Maplify has report time (`created_at`), not sighting time; view already coerces it to `observed_at` | Emit report time as a precise `eventDate` datetime | Emit date-precision eventDate for Maplify and/or note report-vs-sighting in `dataGeneralizations`; document the limitation |
| GAP-B | Maplify (and possibly native) have **no positional accuracy** (`null AS accuracy`) | Default `coordinateUncertaintyInMeters` to 0 (invalid) or a made-up number | **Omit** the field when unknown; document that Maplify accuracy is unavailable |
| GAP-C | Native `public.observations` has **no record-level license**; only per-photo licenses exist | Hard-code one CC license for all native records as if backed by data | Add a `license` column + policy; decide consent/grandfathering for historical records; surface as requirement |
| GAP-D | Third-party (Maplify/Whale Alert) redistribution rights unconfirmed | Assume aggregation is fine and pick CC0 | Confirm rights before publishing; carry structured attribution; possibly ship native-only first |
| GAP-E | `basisOfRecord` not stored anywhere | Leave blank or guess `Occurrence` | Assign `HumanObservation` per source explicitly; document |
| GAP-F | Counts often null; `occurrenceStatus` not modeled | Emit `individualCount=0` / infer status | `occurrenceStatus="present"` always; omit count when null |
| GAP-G | Whale identifiers regex-extracted, unvalidated; organismID deferred | Emit as `organismID`/identity | Keep out of identity terms; if shown, label as unverified in remarks |
| GAP-H | Photos may have **no** license (native/Maplify) | Default to a permissive license | Exclude license-less photos from the Multimedia extension (mirror the iNat `license IS NOT NULL` filter) |

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Hand-concatenate TSV strings instead of a real serializer | Fast to write | Embedded delimiter/newline corrupts archive silently (Pitfall 4) | Never for the public archive |
| Default unknown coordinateUncertainty to 0 | Non-null "clean" column | Invalid per GBIF; claims perfect precision (Pitfall 6) | Never |
| Emit Maplify report time as precise eventDate | Reuses existing view as-is | Asserts false sighting precision to researchers (Pitfall 5) | Never silently; OK only if documented as a known limitation |
| Maintain meta.xml indices by hand | Quick first archive | Drifts from column order → silent corruption (Pitfall 1) | Only for a throwaway prototype, never the shipped generator |
| Overwrite canonical zip directly each night | Simple publish | Torn reads / stale cache / empty-result wipe (Pitfall 14) | Never for a public file; use write-then-swap |
| Hard-code one license for all records | Unblocks export | Misrepresents rights for contributor & third-party data (Pitfalls 11,12) | Never until rights/consent resolved |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| PostGIS → DwC coordinates | Mapping latitude←ST_X / longitude←ST_Y | latitude←ST_Y, longitude←ST_X; unit-test with a known Salish Sea point |
| GBIF Multimedia extension | Linking photos by a re-derived id that drifts from core | Reuse the exact `occurrenceID` string for `coreid`; anti-join check |
| CloudFront in front of S3 | Assuming a new upload is instantly served | Invalidate path or use dated/hashed filenames; flip pointer last |
| Supabase pg_cron / Edge Function export | Materializing full history in memory | Stream via `COPY`; size the runner; empty-result guard |
| `inaturalist.taxa` hierarchy | Emitting scientificName without taxonRank | Always emit `taxonRank` from `taxa.rank`; walk parents for classification |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Full-history regeneration in memory | Cron timeout, OOM | Stream COPY → zip; benchmark | Years of accumulated records |
| Re-running taxon-hierarchy walk per row | Slow generation | Precompute classification per taxon_id (CTE/materialized) | Large occurrence counts |
| No empty-result guard | Good archive overwritten by empty one | Threshold check before publish | Any transient query failure |

## Security / Integrity Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Exposing photo EXIF/precise location in export | Leaks observer/private location (already an EXIF concern per CONCERNS.md) | Strip EXIF; only export intended coordinates; respect generalization |
| Publishing license-less / private records | Republishing data without rights | Filter to licensed records; exclude private; mirror iNat license filter |
| Half-written zip served publicly | Corrupt download, looks like data loss | Atomic write-then-swap + checksum |

## "Looks Done But Isn't" Checklist

- [ ] **meta.xml:** parses against the DwC text-guide schema AND its field indices match emitted column order (round-trip a known record).
- [ ] **coreId join:** every multimedia `coreid` matches a core `id` (anti-join empty); excluded sources filtered from BOTH files.
- [ ] **occurrenceID:** unique in the file; stable scheme documented; not row-position-derived.
- [ ] **eventDate:** ISO-8601, no padding/mismatch issues; Maplify report-time limitation documented, not disguised.
- [ ] **coordinateUncertaintyInMeters:** never 0; omitted when unknown.
- [ ] **geodeticDatum:** present (`EPSG:4326`); lat/lon sign + axis verified by test.
- [ ] **taxonRank + classification:** emitted for every record, including genus/family-only.
- [ ] **occurrenceStatus=present; individualCount** omitted (never 0) when null.
- [ ] **basisOfRecord=HumanObservation** on every record.
- [ ] **license:** dataset license chosen with rights basis; native record-level license gap resolved; license-less photos excluded.
- [ ] **Encoding:** UTF-8 no BOM end-to-end; HTML stripped from body; emoji/accents round-trip.
- [ ] **Nightly publish:** atomic write-then-swap, CloudFront cache handled, empty-result guard, checksum, timezone defined.
- [ ] **EML:** present, with dataset title, license, contact, and the documented temporal/spatial/precision caveats.

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| meta.xml index drift | LOW (pre-publish) / MEDIUM (post) | Regenerate from single field-list source; re-validate |
| coreId orphans | LOW | Add anti-join gate; regenerate |
| Published without third-party rights | HIGH | Pull archive, scope to native-only, obtain permission, re-issue; reputational repair |
| Contributor data published without license/consent | HIGH | Pull/scope archive; add license+consent; re-publish with policy |
| Stale/corrupt zip served | LOW | Invalidate CloudFront; re-run publish with atomic swap |
| Fabricated eventDate already distributed | MEDIUM | Re-issue with date-precision + documented caveat; notify consumers |
