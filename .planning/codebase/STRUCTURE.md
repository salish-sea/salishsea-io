# Codebase Structure

**Analysis Date:** 2026-02-26

## Directory Layout

```
salishsea-io/
├── src/                        # TypeScript source code (Lit components + utilities)
├── supabase/                   # Supabase configuration and migrations
├── infra/                      # Infrastructure as Code (AWS CDK)
├── test/                       # Test data and fixtures
├── htdocs/                     # Static files (legacy/backup)
├── public/                     # Public assets
├── bin/                        # Utility scripts
├── dist/                       # Build output (generated)
├── node_modules/               # Dependencies (generated)
├── index.html                  # HTML entry point
├── package.json                # Node dependencies
├── tsconfig.json               # TypeScript configuration
├── vite.config.js              # Vite build configuration
└── database.types.ts           # Supabase type definitions (generated)
```

## Directory Purposes

**src/**
- Purpose: All TypeScript source code for the web application
- Contains: Lit web components, utilities, types, styling, authentication logic
- Key files: `salish-sea.ts`, `obs-map.ts`, `obs-panel.ts`, `sighting-form.ts`

**supabase/**
- Purpose: Database schema, migrations, and Supabase configuration
- Contains: SQL migration files, edge functions, RLS policies
- Key files: Schema definitions and database setup

**infra/**
- Purpose: Infrastructure as Code for AWS deployment
- Contains: AWS CDK TypeScript stack definitions
- Key files: CDK app configuration

**test/**
- Purpose: Test data, fixtures, and reference files
- Contains: JSON samples, CSV taxonomies, example requests
- Key files: `occurrences.json`, `taxa.csv`, `example-sighting-request.json`

**dist/**
- Purpose: Generated build output (do not commit)
- Contains: Compiled JavaScript, HTML, sourcemaps
- Generated by: `npm run build`

**bin/**
- Purpose: Utility and build scripts
- Contains: CSP hash verification script

## Key File Locations

**Entry Points:**
- `index.html`: Main HTML document; loads main component and Google Sign-In
- `src/salish-sea.ts`: Root web component (customElement 'salish-sea')
- `src/index.css`: Global styles

**Configuration:**
- `tsconfig.json`: Compiler options (ES2023 target, strict mode, decorators)
- `vite.config.js`: Build tool configuration, Sentry plugin
- `package.json`: Dependencies, build scripts, version metadata
- `database.types.ts`: Auto-generated Supabase schema types (do not edit)

**Core Logic:**
- `src/salish-sea.ts`: Main component, state management, event orchestration
- `src/obs-map.ts`: Map initialization, layer management, feature rendering
- `src/obs-panel.ts`: Observation list panel, date selection, edit mode
- `src/sighting-form.ts`: Form for submitting new observations
- `src/segments.ts`: Algorithm for grouping observations into travel chains
- `src/identifiers.ts`: Detection logic for individual/pod identification

**Data Access:**
- `src/supabase.ts`: Supabase client singleton initialization
- `src/identity.ts`: Authentication context and contributor lookup
- `src/occurrence.ts`: Occurrence feature conversion and database queries
- `src/photo-attachment.ts`: Photo upload and EXIF reading

**Styling:**
- `src/style.ts`: OpenLayers feature styling (colors, symbols, text)
- `src/index.css`: Global CSS (fonts, layout)

**Testing:**
- `src/*.test.ts`: Vitest unit tests (co-located with source)
- `test/`: Test data files

## Naming Conventions

**Files:**
- Components: kebab-case with .ts extension (`obs-map.ts`, `sighting-form.ts`)
- Utilities/Services: kebab-case (`supabase.ts`, `photo-attachment.ts`)
- Tests: suffix with `.test.ts` (`segments.test.ts`, `identifiers.test.ts`)
- Assets: descriptive lowercase with extension (`arrow.png`, `hydrophone-default.svg`)

**Directories:**
- lowercase (`src`, `test`, `infra`, `supabase`)
- descriptive names (`node_modules`, `dist`)

**Code:**
- Types: PascalCase (`User`, `Occurrence`, `Segment`, `Contributor`)
- Functions: camelCase (`fetchOccurrences()`, `receiveOccurrences()`, `occurrence2feature()`)
- Constants: UPPER_SNAKE_CASE (rarely used; inline values preferred)
- Private fields: # prefix in classes (`#mapMoveDebounceTimer`, `#handlePopState`)
- Properties: camelCase with @property or @state decorators

**Custom Elements:**
- kebab-case tags (`<salish-sea>`, `<obs-map>`, `<obs-panel>`, `<sighting-form>`)
- Registered via @customElement decorator

## Where to Add New Code

**New Feature (e.g., filter observations):**
- Primary code: `src/` (new component or extend existing like `obs-panel.ts`)
- Tests: Co-located `src/new-feature.test.ts` using Vitest
- Styling: Add styles to component's static styles or extend `src/style.ts` if map-related
- Types: Extend `src/types.ts` if new database schema needed

**New Component/Module:**
- Implementation: `src/my-component.ts` (Lit component) or `src/my-utility.ts` (function)
- Naming: kebab-case file names
- Registration: Add @customElement('my-component') if web component
- Usage: Import and use in parent components or templates

**Utilities:**
- Shared helpers: `src/utils.ts` (contains `compactMap()`)
- Domain-specific: `src/segments.ts`, `src/identifiers.ts` (transformation logic)
- Styling: `src/style.ts` (OpenLayers style functions)
- Constants: `src/constants.ts` (geographic extents, license codes, species speeds)

**Data Types:**
- All types: `src/types.ts` (derive from Supabase schema via `database.types.ts`)
- Context types: Exported from context modules (`src/identity.ts`, `src/map-context.ts`)

## Special Directories

**assets/:**
- Purpose: Embedded images and GeoJSON data files
- Generated: No
- Committed: Yes
- Contents: `arrow.png`, `favicon.ico`, `hydrophone-default.svg`, GeoJSON feature collections

**dist/:**
- Purpose: Built application output
- Generated: Yes (by Vite during `npm run build`)
- Committed: No
- Contents: Minified JS, bundled CSS, HTML, sourcemaps

**supabase/:**
- Purpose: Database schema versioning
- Generated: Migrations auto-generated by Supabase CLI
- Committed: Yes
- Contents: .sql migration files, Edge Function code

**node_modules/:**
- Purpose: Installed npm dependencies
- Generated: Yes (by npm/yarn)
- Committed: No
- Managed by: `package.json` and lockfile

## Architecture Patterns in File Organization

**Lit Component Pattern:**
- Each component is a single `.ts` file with class extending LitElement
- Example: `obs-map.ts` contains class ObsMap with imports, styles, properties, lifecycle hooks
- No separate template files (templates inline via `render()` method using `html` template literal)

**Context Provider Pattern:**
- Context definitions: Separate files (`identity.ts`, `map-context.ts`, `drawing-context.ts`)
- Provider: Root component `salish-sea.ts` uses @provide decorators
- Consumers: Child components use @consume decorators to inject

**Event Delegation Pattern:**
- Root component `salish-sea.ts` listens for custom events ('date-selected', 'focus-occurrence', 'map-move')
- Child components dispatch events via `this.dispatchEvent(new CustomEvent(...))`
- No props-drilling; events bubble to root for state management

**Functional Transformation Pattern:**
- `segments.ts`: Pure functions (occurrences2segments, segment2features) transform data
- `identifiers.ts`: Pure functions (detectIndividuals, detectPod) analyze observations
- No classes/mutation in utility modules

**Styling Strategy:**
- Global styles: `index.css` for layout and fonts
- Component styles: Lit static `css` in each component
- Map/feature styles: Dedicated `style.ts` exporting functions that return OpenLayers Style objects
