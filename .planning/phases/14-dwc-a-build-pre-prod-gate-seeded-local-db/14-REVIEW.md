# Phase 14: Code Review Report

**Reviewed:** 2026-06-21T00:00:00Z
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

Reviewed the Phase 14 CI pre-prod gate changes: a new CI-only SQL fixture
(`supabase/ci-seed.sql`), the `build.yml` edit that applies it via `psql` and
runs the seeded test suite with a step-scoped `SUPABASE_DB_URL`, and the
`guard.test.ts` fix to the DSN-gated DuckDB mock.

I traced the fixture against the actual schema migrations
(`20260619184037_reference_tables.sql`, `20260620000000_resolution_schema.sql`,
`20260621000000_dwc_view_rebuild.sql`, `20260203234153_individuals.sql`) and
the guard implementation (`scripts/dwca/guard.ts`). The mechanics are sound:
taxon `41521` exists, the `create_contributor_on_sign_in` trigger fires on a
plain `INSERT ON auth.users` and supplies the contributor/`user_contributor`
rows the native observation needs, the migration-resolved `DEFAULT`s for
`collection_id`/`provider_id` fire correctly, and the maplify trust/`is_test`/
`number_sighted` filters line up with the three seeded rows. The local-dev DSN
is the well-known Supabase default and is not a credential leak (confirmed,
not flagged).

No blockers. The findings below are robustness and maintainability concerns —
the most important is the fixture's reliance on hardcoded surrogate IDs that
the schema's own migration explicitly warns against (Pitfall 3/4).

## Warnings

### WR-01: Fixture hardcodes `collection_id` surrogate IDs the schema explicitly says not to hardcode

**File:** `supabase/ci-seed.sql:121-122`, `supabase/ci-seed.sql:70`
**Issue:** Row A pins `collection_id = 1` with the comment "orca-network", and
the native observation relies on the comment that "collection_id defaults to 10
(salishsea-direct)". But `public.collections.id` is
`INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY` and the reference seed
(`20260619184037_reference_tables.sql:109-136`) inserts every collection by
slug with **no explicit id**, under `ON CONFLICT (slug) DO NOTHING`. The IDs are
therefore an artifact of insertion order: `orca-network` happens to be row 1 and
`salishsea-direct` happens to be row 10 *today*. The same migration loudly warns
against exactly this coupling ("ON CONFLICT (slug) — natural key, NOT id
(Pitfall 4)", "organization_id resolved via slug subquery ... Pitfall 3"). If
anyone reorders the collections seed or inserts a row above `orca-network` /
`salishsea-direct`, this fixture will silently map Row A to the wrong collection
and Row C's native row to the wrong dataset, breaking the gate's Step 15.5
associated-parties assertion without any error at insert time.
**Fix:** Resolve IDs by slug instead of hardcoding, mirroring the migration's
own pattern:
```sql
-- Row A
... , (SELECT id FROM public.collections WHERE slug = 'orca-network'))
ON CONFLICT (id) DO NOTHING;
```
For the native observation, omit `collection_id` (the DEFAULT already resolves to
`salishsea-direct` by slug) and drop the brittle "defaults to 10" / "defaults to
1" literals from the comments — state the slugs, not the surrogate IDs.

### WR-02: Fixture is not idempotent on re-run, contradicting its own "safe to re-run" intent

**File:** `supabase/ci-seed.sql:32-44`, `supabase/ci-seed.sql:59-97`
**Issue:** The header claims the file is "Safe to re-run ONLY on a freshly-reset
local DB" and applies `ON CONFLICT DO NOTHING` to the Maplify rows as
"belt-and-suspenders", but the `auth.users` INSERT (Section 1) and the
`public.observations` INSERT (inside the `DO $$` block) have **no** conflict
handling. On any second application against a non-reset DB, the `auth.users`
primary-key conflict aborts the whole script under `ON_ERROR_STOP=1`. Because
the Maplify rows guard against conflicts but the native rows do not, the
"belt-and-suspenders" framing is misleading — the file is strictly
single-shot. In CI this is currently fine (`supabase db start` yields a fresh
DB), but the inconsistency is a latent foot-gun for anyone who re-runs the
fixture locally to "top up" a dirty DB.
**Fix:** Either make the native inserts idempotent too —
```sql
INSERT INTO auth.users (...) VALUES (...) ON CONFLICT (id) DO NOTHING;
INSERT INTO public.observations (...) VALUES (...) ON CONFLICT (id) DO NOTHING;
INSERT INTO public.observation_photos (...) VALUES (...)
  ON CONFLICT (observation_id, seq) DO NOTHING;
```
or update the header to state plainly that the fixture must only ever be applied
to a freshly-reset DB and remove the "belt-and-suspenders" wording so it does
not imply re-run safety it does not provide.

### WR-03: `auth.users` direct INSERT omits version-fragile NOT NULL columns

**File:** `supabase/ci-seed.sql:32-44`
**Issue:** The fixture inserts directly into GoTrue's `auth.users` with only
`id, email, created_at, updated_at, raw_user_meta_data`. GoTrue's `auth.users`
schema is owned by the Supabase CLI image and has historically carried
additional NOT-NULL-without-default columns (`aud`, `role`,
`confirmation_token`, `email_change`, `recovery_token`, etc.) that vary across
CLI versions. The workflow pins `supabase/setup-cli@... v1` to version `2.53.6`
(`build.yml:26-28`), so this works today, but a CLI bump could make the bare
INSERT fail. Because `ON_ERROR_STOP=1` is set (`build.yml:31`) the failure would
be loud rather than silent, so this is a robustness/maintainability concern, not
a correctness blocker.
**Fix:** Populate the historically-required columns defensively, or note in the
header comment that the INSERT column set is coupled to CLI version `2.53.6` and
must be re-validated on any `setup-cli` version bump:
```sql
INSERT INTO auth.users (
    instance_id, id, aud, role, email,
    created_at, updated_at, raw_user_meta_data
) VALUES (
    '00000000-0000-0000-0000-000000000000',
    '00000000-0000-0000-0000-000000000001',
    'authenticated', 'authenticated', 'ci-gate@example.com',
    NOW(), NOW(), '{"name": "CI Gate Test"}'::jsonb
);
```

## Info

### IN-01: Header documents a different apply command than the workflow actually uses

**File:** `supabase/ci-seed.sql:4-5`
**Issue:** The header says the fixture is "Applied via:
`supabase db query --local --file supabase/ci-seed.sql`", but `build.yml:30-31`
applies it with `psql "...:54322/postgres" -v ON_ERROR_STOP=1 -f
supabase/ci-seed.sql`. The two are not equivalent (`psql -f` runs the whole
file as one connection with `ON_ERROR_STOP`; `supabase db query` wraps the
Management/CLI path). The stale comment will mislead a future maintainer
debugging a seed failure.
**Fix:** Update the header to show the actual `psql -f ... -v ON_ERROR_STOP=1`
invocation used by `build.yml`, or apply the fixture from the workflow with the
command the comment documents — pick one and keep them in sync.

### IN-02: Comments assert fixed surrogate-ID defaults ("defaults to 10", "defaults to 1")

**File:** `supabase/ci-seed.sql:48-49`, `supabase/ci-seed.sql:69-70`,
`supabase/ci-seed.sql:103`
**Issue:** Multiple comments state the resolved DEFAULTs as concrete integers
("collection_id defaults to 10 (salishsea-direct)", "provider_id defaults to 1
(direct)", "provider_id omitted → DEFAULT 2 (maplify)"). These DEFAULTs are
resolved by slug at migration time (`20260620000000_resolution_schema.sql:197-202`,
`20260619203013_source_table_fk_columns.sql:58-95`), so the integers are
incidental and could drift if collection/provider seed order changes. The
behavior is correct because the column is omitted from the INSERT and the
DEFAULT fires — but the comments encode the same fragile assumption as WR-01.
**Fix:** Describe the DEFAULTs by slug ("DEFAULT resolves to the
`salishsea-direct` collection") rather than by the literal id, to avoid implying
a hardcoded contract.

### IN-03: Test 4 comment references obsolete `mockRestore()` rationale at a stale line

**File:** `scripts/dwca/guard.test.ts:192-199`
**Issue:** The new comment correctly explains why `mockRestore()` is replaced by
`importActual` + `mockImplementation` (the module uses `vi.fn()`, not
`vi.spyOn()`). This is a good fix. Minor note: the delegation binds
`RealDuckDBInstance.create` but the surrounding test still depends on a live DB
(`HAS_DSN`-gated via `test.skip`), so the real DuckDB path only exercises in the
seeded CI job — confirm the CI `npm test` step (`build.yml:42-45`) is the only
place `SUPABASE_DB_URL` is set so this test is never accidentally skipped where
it is expected to run, nor accidentally run where DuckDB's postgres extension is
unavailable. No code change required; this is a coverage observation.
**Fix:** None required. Optionally add a one-line assertion or log in CI that
Test 4 actually ran (not skipped) so a silently-unset `SUPABASE_DB_URL` cannot
quietly drop the only live-DB row-floor test.
