<objective>
Create the v1.3 provenance graph's reference tables — `public.providers`,
`public.organizations`, `public.collections` (with a locked `collection_kind`
enum and nullable `organization_id` FK) — plus a nullable `orcid` column on the
existing `public.contributors`, all in a single idempotent timestamped migration
with the canonical seed embedded in-migration. Then prove it with a psql
assertion snippet that exercises every ROADMAP success criterion, including the
`anon`-role SELECT smoke tests and the write-closed RLS guarantee.

This is the foundation every later v1.3 phase depends on: Phase 10 adds FK
columns on the four source tables referencing these tables; Phase 11 resolves
bracket tags to `collections.slug`; Phase 12 JOINs `public.collections` from the
`dwc.*` views for per-collection attribution.

Purpose: Unblock every downstream FK addition and backfill (Phases 10-12) by
landing the canonical reference data, readable by all PostgREST consumers and
immutable to the public.
Output: One migration file + one assertion snippet; no app code, no config.toml
changes, no new schemas.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anon/authenticated → PostgREST → public.* reference tables | Untrusted public clients reach `public.providers`/`organizations`/`collections` directly via PostgREST. Read must be open (these are canonical public data); write must be closed. |
| service_role / migration → public.* | Trusted writer. `supabase db reset` and future migrations seed/alter these tables; service_role bypasses RLS by design. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Tampering | `public.providers` / `public.organizations` / `public.collections` write path via PostgREST | mitigate | RLS enabled with SELECT-only policies (`FOR SELECT ... USING (true)`); NO INSERT/UPDATE/DELETE policy and NO `GRANT INSERT` for anon/authenticated (D-10, least-privilege skill rule). Acceptance: the assertion snippet runs `SET ROLE anon; INSERT INTO public.providers ...` and confirms it is rejected with `insufficient_privilege`. |
| T-09-02 | Information Disclosure (inverse — data wrongly hidden) | RLS-enabled public table with no matching SELECT policy returns zero rows via PostgREST | mitigate | Every `ENABLE ROW LEVEL SECURITY` is immediately followed by a `CREATE POLICY ... FOR SELECT ... USING (true)` in the same migration (Pitfall 2). Acceptance: `SET ROLE anon; SELECT COUNT(*)` > 0 for all three tables in the snippet. |
| T-09-03 | Information Disclosure | PostgREST over-exposure of new `public` tables | accept | Tables carry only non-sensitive canonical provenance metadata (slugs, names, public URLs, enum kinds); no PII, secrets, or user data. Auto-exposure of `public` is intended (D-01). No mitigation needed beyond confirming no sensitive columns were added. |
| T-09-04 | Tampering (schema integrity) | `aggregator_ingest` re-added to `collection_kind` by this or a future migration | mitigate | Enum value excluded by construction (D-09); SC-3 assertion `PERFORM 'aggregator_ingest'::public.collection_kind` MUST raise `invalid_text_representation`.
