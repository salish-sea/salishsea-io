## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** All three reference tables (`providers`, `organizations`, `collections`) live in the **`public` schema**, alongside `public.contributors`.
- **D-02:** Collections are a **shared registry, not 1:1 with providers.** A collection is reachable by any provider.
- **D-03:** Seed **one baseline collection per non-aggregator provider** — `iNaturalist`, `HappyWhale`, `SalishSea.io Direct`.
- **D-04:** Native (`public.observations`) records that repost Maplify-tagged content resolve to the **same** shared collection; `SalishSea.io Direct` is the fallback for genuinely-original submissions.
- **D-05:** `providers` and `collections` (and `organizations`) carry a serial `id` PK **plus a UNIQUE human-readable `slug`/`code`**. Seed is idempotent via `ON CONFLICT (slug) DO …`.
- **D-06:** Seed un-decoded Maplify acronym tags as **stub collection rows** (`PSWS`, `MCW`, `CWW`, `WSSJI`, `HIWS`, `SBW`, `WA`, `SSCH`, `SA`, `PSWW`, `Bremerton FB group`). Stub = slug from lowercased acronym, name = acronym, organization_id = NULL, kind = NULL.
- **D-07:** Pre-seeding stubs is a deliberate one-time act (NOT auto-create); does not conflict with RESOLVE-04.
- **D-08:** Phase 9 seeds from the point-in-time census in `v1.3-EXECUTIVE-SUMMARY.md §3`. Re-census happens in Phase 11.
- **D-09:** `collections.kind` must be **nullable**. Enum locked to `facebook_group`, `research_dataset`, `acoustic_feed`, `detector`, `direct_app` — NO `aggregator_ingest`.
- **D-10:** Each reference table gets **`ENABLE ROW LEVEL SECURITY` + a permissive `USING (true)` SELECT policy** for `anon` and `authenticated`. No plain GRANT SELECT (that is the `dwc` schema pattern for unexposed schemas).
- **D-11:** Add a **nullable `orcid` column** to the existing `public.contributors`. Do not create a new table. Contributors stay per-provider.

## Summary

Phase 9 is a pure database migration phase: three new reference tables (`providers`, `organizations`, `collections`) plus a nullable column on an existing table (`public.contributors.orcid`), all in a single timestamped SQL migration with idempotent seed data embedded in-migration. No application code changes, no config.toml changes, no new schemas.

The entire shape of this phase is determined by four existing precedents already in the codebase: (1) `public.contributors` defines the PK style (`INTEGER GENERATED BY DEFAULT AS IDENTITY`) and RLS enablement pattern; (2) `20260207002748_user_contributor_rls.sql` shows the exact permissive SELECT policy that makes a `public` table readable by all (`USING (TRUE)` with no role restriction — the policy itself is the access gate); (3) `public.sex` and `public.travel_direction` show the `CREATE TYPE public.<name> AS ENUM (...)` convention; (4) `supabase/snippets/05_dwc_assertions.sql` defines the structure and style for `09_*_assertions.sql`.

The critical RLS insight for this project: `public` tables are PostgREST-exposed by default (`supabase/config.toml` lists `"public"` in `api.schemas`). For PostgREST-exposed tables, RLS policies — not GRANT SELECT — are the access control mechanism. The `dwc` schema uses plain `GRANT SELECT` because it is NOT PostgREST-exposed; `public` tables use RLS policies. This distinction is locked in D-10 and confirmed by existing migration code.

**Primary recommendation:** Write one migration file (`20260619XXXXXX_reference_tables.sql`) that creates the enum type, three reference tables, the RLS policies, the `orcid` column, and the complete idempotent seed — in that dependency order. Then write `09_reference_assertions.sql` mirroring the Phase 5 snippet structure.

### Pattern 1: Table DDL with IDENTITY PK + UNIQUE slug

[VERIFIED: codebase grep — `20260203234153_individuals.sql` line 6 and `20250903172708_initial_schema.sql` line 55]

The project uses `INTEGER GENERATED BY DEFAULT AS IDENTITY` for PKs (not `SERIAL`, not `BIGINT`). `GENERATED BY DEFAULT` (not `GENERATED ALWAYS`) allows explicit ID values during seed — useful for idempotent seed referencing sibling rows by ID. However, since our seed uses slug-based subqueries for FKs (pattern 3 below), `GENERATED BY DEFAULT` vs `GENERATED ALWAYS` is moot here. Mirroring the existing `contributors` and `happywhale.species` convention is the right call.

```sql
-- Source: supabase/migrations/20260203234153_individuals.sql (line 6)
-- and 20250903172708_initial_schema.sql (line 55)
CREATE TABLE public.providers (
  id   INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  slug VARCHAR(50) NOT NULL UNIQUE,
  name VARCHAR(100) NOT NULL
);
```

### Pattern 3: RLS for Public-Reference Tables (Permissive SELECT for All)

[VERIFIED: codebase grep — `20260207002748_user_contributor_rls.sql` and `20250915171505_sighting_policies.sql`]

Two sub-patterns exist in the codebase. For reference tables that ANY user (including anonymous) must read:

```sql
-- Source: supabase/migrations/20260207002748_user_contributor_rls.sql (line 8)
-- Pattern: no TO clause = applies to all roles including anon
ALTER TABLE public.contributors ENABLE ROW LEVEL SECURITY;
CREATE POLICY "View all contributors" ON public.contributors
  FOR SELECT USING (TRUE);
```

Alternatively, with explicit role list (also seen in codebase):

```sql
-- Source: supabase/migrations/20250915171505_sighting_policies.sql (line 40-42)
CREATE POLICY "Observations are visible to everyone."
ON public.observations FOR SELECT
TO anon USING ( true );
```

**Recommendation for Phase 9:** Use the explicit `TO anon, authenticated` form (matching the `taxa` policy at line 60-62 of sighting_policies) because it documents intent clearly. The `success criteria` require `SET ROLE anon; SELECT COUNT(*)` to return > 0 — both patterns satisfy this, but explicit roles are more self-documenting in a reference table context.

```sql
-- Recommended for providers, organizations, collections:
ALTER TABLE public.providers ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Providers are publicly readable"
  ON public.providers FOR SELECT TO anon, authenticated
  USING (true);
```

**Critical distinction from dwc schema:** The `dwc` schema uses `GRANT SELECT ON ALL TABLES IN SCHEMA dwc TO anon, authenticated` (no RLS). That pattern is for schemas NOT exposed via PostgREST. For `public` tables that PostgREST serves, the RLS policy is the gate; a plain GRANT SELECT without RLS policies on an RLS-enabled table would return zero rows (Supabase's default-deny behavior when RLS is enabled with no matching policy). [VERIFIED: STATE.md note — "SELECT grants on new reference tables must be included in the table-creation migration — not a follow-up. Supabase RLS defaults can silently zero-out DwC JOINs."]

**Do NOT add GRANT SELECT for these public tables.** The public schema already has `USAGE` granted to `anon` and `authenticated` by Supabase defaults; RLS policies cover row-level access. Adding `GRANT SELECT` is not wrong but is redundant for PostgREST-exposed tables.

### Pattern 4: Idempotent Seed with ON CONFLICT + Slug-Subquery for FK Resolution

[ASSUMED — standard PostgreSQL pattern; confirmed as the project's chosen approach via D-05/D-06]

```sql
-- Organizations first (no FKs to other new tables)
INSERT INTO public.organizations (slug, name, url, rights_holder_text)
VALUES
  ('orca-network',           'Orca Network',                    'https://orcanetwork.org',           'Orca Network'),
  ('cascadia',               'Cascadia Research Collective',    'https://cascadiaresearch.org',       'Cascadia Research Collective'),
  ('tmmc',                   'The Marine Mammal Center',        'https://marinemammalcenter.org',     'The Marine Mammal Center'),
  ('mbari',                  'Monterey Bay Aquarium Research Institute', 'https://mbari.org',         'MBARI'),
  ('orcasound',              'Orcasound',                       'https://orcasound.net',              'Orcasound')
ON CONFLICT (slug) DO NOTHING;

-- Collections referencing organizations by slug subquery
INSERT INTO public.collections (slug, name, kind, organization_id)
VALUES
  ('orca-network',     'Orca Network',                    'facebook_group',   (SELECT id FROM public.organizations WHERE slug = 'orca-network')),
  ('cascadia',         'Cascadia Research Collective',    'research_dataset', (SELECT id FROM public.organizations WHERE slug = 'cascadia')),
  ...
  -- Acronym stubs: organization_id NULL, kind NULL
  ('psws',             'PSWS',                            NULL,               NULL),
  ('mcw',              'MCW',                             NULL,               NULL)
ON CONFLICT (slug) DO NOTHING;
```

`DO NOTHING` is correct here (not `DO UPDATE`): these are immutable reference rows. If a row already exists with the right slug, we keep the existing data rather than overwriting it (protects against re-running the migration accidentally in an unusual recovery scenario).

### Pattern 5: ALTER TABLE ADD COLUMN IF NOT EXISTS

[VERIFIED: PostgreSQL docs — IF NOT EXISTS on ADD COLUMN is supported since PostgreSQL 9.6]

```sql
ALTER TABLE public.contributors
  ADD COLUMN IF NOT EXISTS orcid TEXT;
```

`IF NOT EXISTS` makes the migration safe to apply even if the column was added manually in a dev environment. This is the idempotent form. No NOT NULL constraint — values are populated later (D-11).

### Anti-Patterns to Avoid

- **Putting seed data in `supabase/seed.sql`:** `seed.sql` is dev-fetch data, not canonical reference data. It is reset-vulnerable in production context. Canonical provenance data belongs in the migration. [VERIFIED: CONTEXT.md Claude's Discretion + STATE.md note]
- **Creating the enum with `aggregator_ingest`:** Enum values cannot be removed in PostgreSQL without recreating the type. Adding `aggregator_ingest` now would require a future destructive migration. It is explicitly out of scope (D-09). [ASSUMED: standard PostgreSQL behavior — ALTER TYPE ... DROP VALUE is not supported]
- **Using `DO UPDATE SET name = EXCLUDED.name`:** Reference rows should be stable. `DO NOTHING` is the correct conflict strategy for seed rows that represent facts (not mutable data). `DO UPDATE` would silently overwrite manual corrections made after initial seed.
- **Using GRANT SELECT without RLS policies:** On an RLS-enabled public table, GRANT SELECT alone is not sufficient — row-level access requires at least one matching SELECT policy. With RLS enabled and no policy, all queries return zero rows (Supabase default-deny). [VERIFIED: STATE.md migration note]
- **Adding a `kind` column to `providers`:** The four providers are a fixed enumeration; provider type adds complexity without payoff. The slug distinguishes them. Claude's Discretion — recommendation: no `kind` column on `providers`.
- **Seeding collections before organizations:** The `organization_id` FK subquery `(SELECT id FROM organizations WHERE slug = '...')` must resolve, which requires organizations to exist. Order within the migration file matters; organizations must come before collections.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Idempotent seed | Custom upsert logic with CASE/IF blocks | `INSERT … ON CONFLICT (slug) DO NOTHING` | PostgreSQL native; exactly one statement per conflict target |
| RLS "public read" | Application-layer filtering | `ENABLE ROW LEVEL SECURITY` + `CREATE POLICY … USING (true)` | Database-enforced; survives code bypasses; Supabase linter requires it for public tables |
| ORCID format validation | Custom regex CHECK constraint this phase | Nullable TEXT column with CHECK deferred to when values are populated | ORCID population is out of scope; adding a CHECK now risks blocking the eventual populate migration if formatting edge cases exist |

## Column Design (Claude's Discretion Recommendations)

### `providers` columns

```sql
CREATE TABLE public.providers (
  id   INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  slug VARCHAR(50)  NOT NULL UNIQUE,
  name VARCHAR(100) NOT NULL
);
```

No `kind` column recommended — four fixed rows, slug is sufficient for identity. Phase 11 resolver is TypeScript code that maps slug to behavior, not a DB-driven dispatch table.

### `organizations` columns

```sql
CREATE TABLE public.organizations (
  id                 INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  slug               VARCHAR(50)   NOT NULL UNIQUE,
  name               VARCHAR(200)  NOT NULL,
  url                TEXT          NOT NULL,
  rights_holder_text VARCHAR(200)  NOT NULL
);
```

`url NOT NULL` is required by ORG-01 ("non-null url"). `rights_holder_text` is the display string used in EML `associatedParty` (Phase 12); it is typically the same as `name` but stored separately so it can diverge (e.g., abbreviations). All five seeded organizations have known URLs. [ASSUMED: column naming `rights_holder_text` — planner may adjust; the conceptual requirement is ORG-01]

### `collections` columns

```sql
CREATE TABLE public.collections (
  id              INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  slug            VARCHAR(100)         NOT NULL UNIQUE,
  name            VARCHAR(200)         NOT NULL,
  kind            public.collection_kind,          -- nullable: stubs have NULL kind
  organization_id INTEGER REFERENCES public.organizations(id)  -- nullable: stubs + Whale Alert
);
```

No FK to `providers` — this is the shared-registry design (D-02). Phase 10 adds `collection_id` on each source table, which is the per-record provider/collection association.

### `orcid` column on `contributors`

```sql
ALTER TABLE public.contributors
  ADD COLUMN IF NOT EXISTS orcid TEXT;
```

**Type: `TEXT`** — not a domain/CHECK constraint for ORCID format. Rationale: ORCID population is Phase 12+ work; adding a CHECK constraint now creates a risk that values imported in a future migration fail the constraint due to edge cases (e.g., hyphen normalization, `https://orcid.org/` prefix vs. bare `0000-...` format). A CHECK can be added once the actual population pattern is known. [ASSUMED: deferring CHECK — this is a judgment call the planner should confirm with user if they have strong format-validation preference]

**No index needed at this phase.** 28 native contributors is below any threshold where a btree index on an infrequently-queried nullable column provides measurable benefit.

## Common Pitfalls

### Pitfall 1: Enum Value Immutability

**What goes wrong:** Someone adds `aggregator_ingest` to the `collection_kind` enum "just in case", then it needs to be removed. PostgreSQL has no `ALTER TYPE ... DROP VALUE`. The only fix is `CREATE TYPE` with the new value set + `ALTER TABLE ... ALTER COLUMN TYPE` + `DROP TYPE` on the old type — a multi-step migration with a table lock.

**Why it happens:** Desire to future-proof. The design decision (D-09) already made the call: no `aggregator_ingest`. Provider is a separate concept.

**How to avoid:** Enumerate exactly the five locked values. Add a comment in the migration referencing D-09.

**Warning signs:** Any PR adding `aggregator_ingest` to the enum or to the `collections` seed.

### Pitfall 2: RLS Enabled + No Policy = Invisible Table

**What goes wrong:** `ENABLE ROW LEVEL SECURITY` is applied but no `CREATE POLICY` follows. `SELECT * FROM providers` returns 0 rows even with a valid database connection. Supabase's default-deny RLS behavior means no policy = no access.

**Why it happens:** The `dwc` schema uses `GRANT SELECT` without RLS and works — developer cargo-cults that pattern to a public table.

**How to avoid:** For every `ENABLE ROW LEVEL SECURITY`, immediately follow with `CREATE POLICY ... USING (true)` in the same migration file. The assertions snippet `SET ROLE anon; SELECT COUNT(*) FROM providers` will catch this during local verification.

**Warning signs:** `SELECT COUNT(*) FROM providers` returns 0 after `supabase db reset`.

### Pitfall 3: Seed Ordering — Collections Before Organizations

**What goes wrong:** The `organization_id` subquery in the collections INSERT uses `(SELECT id FROM organizations WHERE slug = '...')`. If collections are seeded before organizations, this subquery returns NULL even for named collections that should have a non-null `organization_id`.

**Why it happens:** Copy-paste ordering or a migration split across multiple files where ordering isn't obvious.

**How to avoid:** In a single-file migration, declare DDL and seed in dependency order: (1) enum type, (2) providers DDL + seed, (3) organizations DDL + seed, (4) collections DDL + seed, (5) `orcid` ALTER TABLE. The assertions snippet checks that named collections have non-null `organization_id`, which catches ordering bugs.

**Warning signs:** All collection `organization_id` values are NULL after reset, even for named collections.

### Pitfall 4: Wrong Conflict Target on INSERT … ON CONFLICT

**What goes wrong:** `ON CONFLICT (id) DO NOTHING` instead of `ON CONFLICT (slug) DO NOTHING`. The `id` is IDENTITY-generated and will never conflict during a clean `db reset + seed`. The conflict target must be the UNIQUE constraint column (`slug`), not the PK.

**Why it happens:** Confusion between conflict on the natural key vs. synthetic PK.

**How to avoid:** Always specify `ON CONFLICT (slug) DO NOTHING`. The UNIQUE constraint on slug is what makes the seed idempotent across re-runs.

### Pitfall 5: PostgREST Exposure for `public` Tables — No config.toml Changes Needed

**What goes wrong:** Developer adds `public` to `api.schemas` again, or adds a new schema entry, thinking public tables need extra configuration.

**Why it happens:** Overreading the `dwc` schema's comments about not being in `api.schemas`.

**How to avoid:** `supabase/config.toml` already lists `"public"` in `api.schemas`. New tables in `public` are auto-exposed via PostgREST with zero config changes. The only gate is RLS. Do not modify `config.toml` in this phase.

### Pitfall 6: Phase 12 DwC JOIN Silently Returns NULL

**What goes wrong:** Phase 12's `dwc._maplify_occurrences` adds a JOIN on `collections.slug` or `collections.id`. If the `SELECT` policy for `dwc` views' service-role is not already satisfied — or if the collections seed has wrong slugs — the JOIN silently returns NULL for the new attribution columns.

**Why it happens:** The `dwc` schema has separate `GRANT SELECT` for its views. But the JOIN target (`public.collections`) relies on RLS. The service role (`postgres`) bypasses RLS by default in Postgres, so this is actually safe — but the `anon` role in the assertions snippet does not bypass RLS. The policy must exist.

**How to avoid:** The `USING (true)` RLS policy covers this. Run the assertions to confirm anon-role access works. Document that Phase 12 will JOIN `public.collections` from within `dwc.*` views — the service-role that runs the DuckDB ATTACH reads `dwc.*` views which in turn JOIN `public.collections`; since the service-role bypasses RLS, this works. The anon-role assertions in `09_reference_assertions.sql` verify the PostgREST path.

### Pitfall 7: Acronym Stub Slug Collision with Future Named Collections

**What goes wrong:** An acronym stub is seeded with slug `wa`, then Phase 11 decodes `WA` as "Whale Alert (Alaska)". But `whale-alert-alaska` is already seeded as a named collection. The Phase 11 migration now must UPDATE the `wa` stub row AND ensure no FK references point to it using the old slug.

**Why it happens:** The `WA` bracket tag is ambiguous, and the seeding in Phase 9 can't resolve it.

**How to avoid:** This is inherent in the design (D-06/D-08). Document the stub explicitly. Phase 11's task is to UPDATE the stub rows' `name`, `kind`, and `organization_id` columns. No slug change needed (slugs are stable natural keys). The Phase 11 plan must include an UPDATE on stub rows, not a new INSERT.
