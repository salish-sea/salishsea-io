<objective>
Add the four per-sighting provenance linkage columns — `provider_id`,
`collection_id`, `contributor_id`, `source_url` — to all four source tables
(`public.observations`, `maplify.sightings`, `inaturalist.observations`,
`happywhale.encounters`); index `collection_id` on the two exported tables
(`public.observations`, `maplify.sightings`); and perform the deterministic
backfill that is fully knowable now (`provider_id` everywhere by provider slug,
`source_url` on native/iNat/HappyWhale). The columns are left ready to receive
the Phase 11 resolver backfill (`collection_id`, `contributor_id`, Maplify
`source_url`) without any NOT NULL constraint on `collection_id`.

Purpose: This unblocks the Phase 11 resolution/backfill and the Phase 12 DwC
view rebuild, which JOIN on these FKs. It wires the schema and the deterministic
data — not the comment-parsing resolver (Phase 11) and not the ingest RPCs (D-14,
forward-population is handled by the generated `source_url` columns and the
migration-resolved `provider_id` DEFAULT).

Output: ONE timestamped migration
`supabase/migrations/20260619203013_source_table_fk_columns.sql` and ONE
assertion snippet `supabase/snippets/10_fk_columns_assertions.sql`, verified by
`supabase db reset` + psql exit 0. Implements LINK-01, LINK-02, LINK-03.
</objective>

generated expression is legal and makes every HW row's `source_url` non-null by
construction. The `/encounter/{id}` form is forbidden.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| migration SQL → Postgres catalog | DDL runs as the migration role (superuser-equivalent) at apply time. No untrusted input — the migration is in-repo SQL; the only interpolated value in dynamic SQL is a server-side `SELECT id` integer. |
| ingest RPC → source tables (existing, unchanged) | Pre-existing write path. Not modified this phase (D-14); new columns inherit it via DEFAULT (provider_id) + GENERATED (source_url). No new write surface. |
| anon/authenticated → public.observations (existing RLS) | Pre-existing read/write boundary, governed by existing policies. New columns add no new policy and are read-only provenance. |

No SQL-injection surface. ASVS V5: input validation is structurally N/A here. |

The new columns inherit the table's existing anon-SELECT-all policy — no new disclosure beyond what the table already permits. No new SELECT policy needed. |
| T-10-03 | Elevation of Privilege | Accidental GRANT/REVOKE or RLS policy change loosening write access on a source table | mitigate | The migration adds NO GRANT/REVOKE and NO RLS policy.
