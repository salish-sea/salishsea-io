    Create `scripts/dwca/assertions.ts` exporting:

    (a) `export interface PgColumn { readonly name: string; readonly ordinal: number; }` — the DuckDB-side representation of one view column. `ordinal` is 1-based (matches `information_schema.columns.ordinal_position`).

    (b) `export class AlignmentError extends Error { constructor(message: string, public readonly table: string, public readonly diff: readonly string[]) { super(message); this.name = 'AlignmentError'; } }`.

    (c) `export function assertFieldAlignment(pgCols: readonly PgColumn[], tsFields: readonly { readonly name: string }[], table: string): void` — the F-02 guard.

    Implementation requirements for `assertFieldAlignment`:
    - Loop `i` from `0` to `max(pgCols.length, tsFields.length) - 1`.
    - At each index, compare `pgCols[i]?.name` vs `tsFields[i]?.name`. Collect mismatch strings:
      * Missing from PG: `[+${i}] TS array has "${tsFields[i].name}" but view has no column at ordinal ${i + 1}` (use the 1-based pg ordinal for human readability).
      * Missing from TS: `[-${i}] View has "${pgCols[i].name}" at ordinal ${pgCols[i].ordinal} but TS array ends`.
      * Name mismatch: `[~${i}] TS expects "${tsFields[i].name}" but view has "${pgCols[i].name}" at ordinal ${pgCols[i].ordinal}`.
    - If the collected diff array is empty, return silently.
    - If non-empty, build a message: `Field alignment mismatch for dwc.${table}:\n${diff.join('\n')}` and throw `new AlignmentError(message, table, diff)`. The caller (build.ts) catches and exits non-zero; tests catch and assert structure.
    - Do NOT call `process.exit` directly — that prevents Vitest from observing the failure. The error-throwing approach lets tests assert.

    (d) `export async function assertNonZeroRows(conn: { runAndReadAll: (sql: string) => Promise<{ getRowObjects: () => Record<string, unknown>[] }> }, fullyQualifiedTable: string): Promise<bigint>` — runs `SELECT COUNT(*) AS n FROM ${fullyQualifiedTable}` and throws `Error('Empty result: ' + fullyQualifiedTable)` if the count is 0n. Returns the count as bigint.
    - Note: the `conn` parameter is structurally typed (not imported from `@duckdb/node-api`) so tests can pass a mock. This keeps `assertions.ts` pure and DuckDB-agnostic.

    (e) `export async function assertNoZeroByteFile(path: string): Promise<void>` — uses `fs/promises` `stat`; throws `Error('Zero-byte file: ' + path)` if `size === 0`. Returns void on success.

    Create `scripts/dwca/assertions.test.ts` covering:
    - `assertFieldAlignment` happy path: arrays match exactly → no throw.
    - Extra TS entry: `tsFields = [{name:'a'},{name:'b'},{name:'c'}]`, `pgCols = [{name:'a',ordinal:1},{name:'b',ordinal:2}]` → throws `AlignmentError` whose `diff` contains a string matching `/\[\+2\].*"c"/` (TS has c at index 2; pg ends).
    - Extra PG column: reverse of above → diff contains `/\[-2\].*"c"/`.
    - Reorder mismatch: `tsFields = [{name:'a'},{name:'b'}]`, `pgCols = [{name:'a',ordinal:1},{name:'x',ordinal:2}]` → diff contains `/\[~1\].*"b".*"x"/`.
    - `AlignmentError` carries `table` and `diff` properties (instance check).
    - `assertNonZeroRows` with mock conn returning `[{n: 5n}]` → resolves to `5n`.
    - `assertNonZeroRows` with mock conn returning `[{n: 0n}]` → rejects with message containing `'Empty result'` and the table name.
    - `assertNoZeroByteFile` with a temp file written via `fs.writeFile` containing `'hello'` → resolves.
    - `assertNoZeroByteFile` with a temp file containing `''` → rejects with message containing `'Zero-byte file'`.
    - Temp files use `os.tmpdir() + '/' + 'dwca-test-' + crypto.randomUUID()` and are cleaned up in `afterEach`.

    Use Vitest `describe`/`test`/`afterEach`. Do NOT import `@duckdb/node-api` in either the source or the test — structural typing keeps the test pure.

    Create `scripts/dwca/zip.ts` exporting:

    (a) `export const FIXED_MTIME = new Date('2000-01-01T00:00:00Z') as const;`

    (b) `export interface ZipEntry { readonly name: string; readonly content: Buffer; }`.

    (c) `export async function writeZip(outPath: string, files: readonly ZipEntry[]): Promise<void>`.

    Implementation requirements:
    - `import yazl from 'yazl';` — yazl exports a default object containing `ZipFile`. Use `new yazl.ZipFile()`.
    - For each `file` in `files`:
      * Validate `file.name`: reject if empty, contains `..`, contains `\0`, starts with `/`, or starts with `\`. Throw `Error('Invalid zip entry name: ' + file.name)`.
      * Call `zip.addBuffer(file.content, file.name, { mtime: FIXED_MTIME, compress: true })`.
    - After adding all files, call `zip.end()`.
    - Stream-pipe `zip.outputStream` to `createWriteStream(outPath)` using `pipeline` from `node:stream/promises`. Await the pipeline.
    - Ensure the parent dir of `outPath` exists: `await fs.mkdir(path.dirname(outPath), { recursive: true });` BEFORE opening the write stream.
    - Return void on success; rethrow any pipeline / mkdir error.

    Create `scripts/dwca/zip.test.ts` covering:
    - Smoke: write a 2-entry zip to a tmp path with names `'meta.xml'` and `'eml.xml'` and `Buffer.from('<?xml ...?>')` content. Assert the file exists and is `>= 50` bytes.
    - Determinism: write the same 4-entry zip twice (different tmp paths). Read both back via `fs.readFile`. Their bytes must be byte-for-byte identical. (Fixed mtime + fixed entry order ⇒ identical zip bytes is RESEARCH §T6's claim — verify it.)
    - Entry order preservation: write a zip with entries `['a.txt', 'b.txt', 'c.txt']`. Read back the central directory file order using `yauzl` if available, OR parse the zip bytes by spec (look for `PK\x03\x04` local file header signatures; the first three `name` fields after each signature must be `a.txt`, `b.txt`, `c.txt`). Since `yauzl` is NOT a project dep, use a regex/Buffer indexOf approach against the file names in the order they appear in raw bytes.
    - Path-traversal rejection: each of `''`, `'../escape.txt'`, `'subdir/../../escape.txt'`, `'/absolute.txt'`, `'\\backslash.txt'`, `'has\0null.txt'` (null byte) — calling `writeZip(tmp, [{name, content: Buffer.from('x')}])` rejects with an error message containing `'Invalid zip entry name'`.
    - Parent dir creation: write to `outPath = tmpDir + '/nested/subdir/test.zip'` (the `nested/subdir` does not exist). Confirm the zip is created and `nested/subdir` is created. After the test, recursively clean up the tmpDir.
    - All temp paths use `os.tmpdir() + '/' + 'dwca-zip-test-' + crypto.randomUUID()` and are cleaned in `afterEach`.

    Do NOT add a test that round-trips the actual `meta.xml` / `eml.xml` content — that integration belongs to Plan 06's `build.test.ts` and runs after `build.ts`.

| Boundary | Description |
|----------|-------------|

| `ZipEntry.name` → yazl ZipFile entry name | hardcoded by build.ts today but defended against path-traversal as belt-and-suspenders |
| `outPath` → fs.createWriteStream | comes from build.ts (hardcoded `dist/dwca/...`); no user input |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04-DRIFT | Tampering | `dwc.occurrences` view drifts; build silently emits wrong meta.xml | mitigate | `assertFieldAlignment` is THE F-02 guard; build.ts calls it before any COPY. Plan 02's unit tests are the second guardrail (catch fields.ts drift in isolation). Plan 06's integration test is the third (round-trip parse). |
| T-06-04-PATH | Tampering | path-traversal in zip entry names | mitigate | `writeZip` validates names against six categories: empty, `..`, NUL, leading `/`, leading `\`. Even though Plan 05's caller passes hardcoded names, the function is defense-in-depth. |
| T-06-04-EMPTY | Denial of service | Build silently produces an empty zip (zero rows in `dwc.occurrences`) | mitigate | `assertNonZeroRows` and `assertNoZeroByteFile` together ensure Phase 7 never publishes a manifestly-broken archive. Plan 05's `build.ts` calls both. |

Build.ts (Plan 05) is responsible for never logging the DSN. |
| T-06-04-ZIPBOMB | Denial of service | malicious zip content | accept | All content originates from DuckDB COPY output of trusted Postgres views; no untrusted upload path exists. |
