> start of the milestone; some details here were superseded during execution
> (e.g. organizations map to EML `associatedParty`, not `samplingProtocol`;

*Date: 2026-06-19. Grounded in a read-only production sample (project ref
`grztmjpzamcxlzecmqca`). Counts are point-in-time and will drift.*

This milestone formalizes the **provenance graph** behind every sighting so
attribution is correct — both inside the app (community + observer credit) and
in the DarwinCore Archive export. Today ~5,500 aggregated records collapse into a
single "Whale Alert / Maplify" bucket; this fixes that.

---

## 1. Terminology — four independent concepts

| Concept | Question it answers | Cardinality | DwC-A mapping |
|---------|---------------------|-------------|---------------|
| **Provider** | *How did this record reach us?* (the ingest API/pipeline) | one per sighting | none directly — internal provenance only |
| **Collection** | *What channel did the observation come through?* (the venue) | one per sighting; **stable** across providers | → `datasetName` (`"SalishSea.io — {collection}"`) |
| **Organization** | *What institution backs the channel?* | nullable, via collection | supplementary EML metadata (`samplingProtocol` / `bibliographicCitation`), **not** `institutionCode` |
| **Contributor** | *Who observed it?* | nullable, per sighting | → `recordedBy` (and `recordedByID` where stable) |

Plus one fixed fact: **SalishSea.io is the aggregator** — `institutionCode =
"SalishSea"`, `rightsHolder = "SalishSea.io"` on every exported row. This is the
dominant pattern among aggregators (Happywhale→OBIS-SEAMAP, iNaturalist, eBird).

**Why provider ≠ collection (the correction to the original design):** in
`Orca Network FB Group → Maplify/conserve.io → SalishSea.io`, *Maplify is the
provider*, *Orca Network FB Group is the collection*. If that same FB group were
entered directly into SalishSea.io, the collection would be unchanged — only the
provider flips. So provider is per-record provenance on the sighting, never a
property of the collection. This kills the `aggregator_ingest` value that the
999.1 note had in `collections.kind`.

## 3. Collections — specific instances (Maplify, prod)

Maplify carries collections via **two signals** in `comments`:

**(a) Leading bracket tag** `[Tag]`:

| Tag | Rows | Resolves to collection / note |
|-----|-----:|-------------------------------|
| `Orca Network` (+ 4 typo variants: `" Orca Network"`, `"Orca Networ"`, `"Orca Networks"`, `"Orca Neteork"`) | 2,252 (+4) | Orca Network FB group |
| `PSWS` | 31 | acronym — resolve during backfill |
| `MCW` | 12 | acronym — resolve |
| `CWW` | 10 | acronym — resolve |
| `WSSJI` | 5 | acronym — resolve |
| `Orcasound` | 3 | Orcasound (community/acoustic) |
| `HIWS` | 3 | acronym — resolve |
| `SBW` | 2 | acronym — resolve |
| `WA` | 2 | acronym — resolve |
| `MBARI` | 2 | Monterey Bay Aquarium Research Institute |
| `SSCH`, `SA`, `PSWW`, `Bremerton FB group`, `[NULL]` | 1 each / 8 empty | one-offs + 8 empty brackets |

The four Orca Network misspellings are the canonical example of why backfill is a
**human-eyeballed exact-match dictionary**, not fuzzy code.

**(b) Trailing "Submitted by a … Trusted Observer" line** (names an org, not a
person — so it yields a *collection/org*, never a *contributor*):

| Attribution | Rows | Collection |
|-------------|-----:|-----------|
| Cascadia Trusted Observer | 1,994 + 20 | Cascadia Research Collective |
| Whale Alert Global Trusted Observer | 801 | Whale Alert (Global) |
| Whale Alert Alaska Trusted Observer | 621 + 3 | Whale Alert (Alaska) |
| TMMC Trusted Observer | 78 + 25 | The Marine Mammal Center |

> **Design correction:** the 999.1 note said to *drop* trailing "Submitted by …"
> lines that name no person. They should be dropped for **contributor**
> extraction, but they are the **only** collection signal for ~2,740 Cascadia /
> Whale Alert / TMMC rows — so they must be *used* for collection/org resolution.
> Collection resolution = bracket tag **or** trailing attribution **or** the
> structured `maplify.sightings.source` code.

Other providers' collections (initial): iNaturalist → one collection
("iNaturalist"); HappyWhale → one collection ("HappyWhale"); Direct → "SalishSea.io
direct submissions". Finer granularity (per iNat project, per HappyWhale org) is
an open question for discuss-phase.

### `source_url` as a resolution signal (Layer 1)

`source_url` is first-class this milestone and the **preferred** collection/provider
signal where present. A URL-pattern registry maps domain/path → provider +
collection:

| URL pattern | Provider | Collection |
|-------------|----------|-----------|
| `inaturalist.org/observations/*` | iNaturalist | iNaturalist (`inaturalist.observations.uri` already populated) |
| `happywhale.com/*` | HappyWhale | HappyWhale (derivable from encounter id) |
| `facebook.com/groups/{slug}/*` | (depends) | that group's collection (future FB ingests) |
| SalishSea.io permalink | Direct | SalishSea.io direct (`public.observations.url`) |

**Resolution order:** `source_url` pattern → leading bracket tag → trailing
attribution → structured `maplify.sightings.source` code → NULL.

> **Reality check:** Maplify records — the milestone's main attribution mess —
> carry **no record permalink today** (only `photo_url`; no `http` in comments).
> So URL resolution shines for iNat + native immediately and for *future* FB
> ingests that capture the post URL, but bracket-tag/attribution parsing is still
> required to backfill the existing Maplify rows.

**Layer 2 (deriving a whole occurrence from a pasted `source_url`) is out of
scope** — seeded at
[seeds/url-to-occurrence-importer.md](seeds/url-to-occurrence-importer.md).

## 5. What makes it into the DarwinCore Archive

**Exported (attribution improves):**

| Source | Today | After v1.3 |
|--------|-------|-----------|
| Maplify (~5,532 rows surviving export filters) | one bucket: `datasetName` = `rightsHolder` = "Whale Alert / Maplify"; `recordedBy` = opaque codes (`whalealertoa`, `cascadiaWebMap`, `farallon`) | per-collection `datasetName` ("SalishSea.io — Orca Network", "…— Cascadia", "…— Whale Alert Global", "…— TMMC", …); `institutionCode="SalishSea"`; `rightsHolder="SalishSea.io"` |
| Native (436 rows) | `rightsHolder` = the *person* (Scott Veirs 350, Serena Tierra 68, …); no `institutionCode` | `rightsHolder="SalishSea.io"`, `recordedBy` = the person, `institutionCode="SalishSea"`, `datasetName` = "SalishSea.io — Direct" |

**Not exported (modeled internally only — SRC-01 unchanged):**

- iNaturalist (8,759) and HappyWhale (5,601) — both already publish to GBIF via
  their own canonical datasets; exporting them from SalishSea.io would duplicate
  records. They get providers/collections/contributors in our DB for UI credit,
  filtering, and future-proofing, but emit nothing to the archive this milestone.

**Net export change:** the ~5,968 exported rows go from 1 opaque dataset bucket
to ~10+ correctly-attributed per-collection datasets, gain an `institutionCode`,
and stop conflating `rightsHolder` with the observer.

## 6. Concepts that do *not* enter the archive

- **Provider** — internal provenance; GBIF treats SalishSea.io as the publisher.
- **Organization** — supplementary EML metadata at most, never `institutionCode`.
- **iNaturalist / HappyWhale records** — excluded per SRC-01.
- **HappyWhale individuals (whales)** — `organismID` is deferred to v2 (INDIV-01).
